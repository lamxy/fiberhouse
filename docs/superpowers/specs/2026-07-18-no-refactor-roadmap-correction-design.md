# 无重构优化路线纠偏设计

> 日期：2026-07-18
> 基线：`main@1384918`
> 状态：设计边界已确认，等待书面规格审阅

## 背景

当前活动优化清单把后续 P1 工作描述为统一运行入口、统一关闭链、
GlobalManager owner/locator 决策和 v1 API 迁移。这些建议会改变公共接口、
跨组件调用关系或代码组织，超出了“在原架构和实现上微调、修复明确不足”的
约束。

本设计撤销这些大范围方向。后续优化不再以统一架构、补充抽象或迁移 API 为
目标，只接受证据明确、边界局部、可独立验证的缺陷修复。

## 硬约束

后续优化必须同时满足以下规则：

1. 不新增 `Run(ctx context.Context) error` 或任何等价运行入口。
2. 不为统一 Fiber、Gin、CLI 或 task 生命周期而修改现有公共接口、方法签名或
   调用链。
3. 不新增 shutdown registry、closer stack、生命周期协调器或同类基础设施。
4. 不拆分 GlobalManager owner/locator，不以完整状态机重写为任务目标。
5. 不迁移公开 package 到 `internal/`，不新增替代性的默认装配 API，不以 API
   治理为由推动结构调整。
6. 不顺带整理、抽象、改名、搬迁文件或统一相似实现。
7. 无法在原函数、原接口和原调用路径内安全修复的问题，只记录为限制，不实施。

## 历史文档处理

已经完成并提交的分析、设计、计划和执行记录保持原样，作为当时决策与执行的
审计材料，不追溯改写。

本次纠偏只修改仍会驱动未来工作的活动清单：
`.codegraph-qa-out/readme-current-status-optimization-todo.md`。

  `.codegraph-qa-out/readme-current-status-analysis-2026-07-17.md`；

## 活动清单重新分类

### 取消

以下内容从活动任务中删除，不替换为另一种架构方案：

- Web/CLI 新运行入口及内部调用迁移；
- shutdown registry/closer stack 和统一关闭链；
- Fiber、Gin、CLI 生命周期统一；
- GlobalManager owner/locator 拆分；
- GlobalManager 完整状态机重写；
- 公开 package 迁入 `internal/`；
- 新默认装配入口和跨 package API 治理工程。

### 保留为事实，不作为重构任务

以下已知限制仍可留在状态说明中，但不能据此推导大范围改造：

- 某些现有入口会记录、吞掉或以 panic/fatal 处理错误；
- 某些资源没有完整关闭或外部集成测试；
- GlobalManager 的引用存活期、旧实例退役和删除行为存在已记录限制；
- Fiber、Gin、CLI 和 task 当前实现并不完全一致。

这些限制只有在发现一个可复现、可局部修复的问题时，才可以拆成独立候选任务。

### 允许的候选任务

候选任务必须是以下形态：

1. 指向一个具体函数或一条短调用路径；
2. 有当前错误行为、失败测试或静态证据；
3. 修复不改变公开签名，不增加跨组件抽象；
4. 修改文件和影响面可以在任务开始前明确列出；
5. 能用聚焦测试证明红绿过程，并用相关包测试和 race 验证；
6. 一次只修复一个缺陷，不夹带相邻清理。

例如，现有 Gin TLS 配置与实际 serve 路径不一致，可以作为后续调查候选；但只有
在确认能沿用当前 `AppCoreRun` 和现有配置结构局部修复后，才允许形成实施计划。
调查本身不授权修改代码。

## 活动清单的新结构

活动清单后续改写为：

1. 已完成事实与执行记录；
2. 无重构硬约束；
3. 未解决限制；
4. 局部缺陷候选及其证据；
5. 候选任务准入门槛；
6. 推荐顺序：先调查、再单项设计、用户审批、独立工作树实施和验证。

不再按“统一生命周期”“完整状态机”“API 治理”等架构主题组织实施路线。

## 后续工作流

每个候选问题独立执行以下流程：

1. 使用 CodeGraph 和当前源码确认真实调用路径；
2. 用测试、race、vet 或最小复现确认问题存在；
3. 给出“不修改”和最多两个局部修复方案；
4. 明确证明方案不改变公开接口和架构；
5. 用户批准后，在新的 git worktree 中按 TDD 实施；
6. 独立审查修改范围，确认没有重构或顺带改动；
7. 用户审核后才同步到 `main`。

如果第 3 或第 4 步无法成立，该候选停止，只保留限制记录。
